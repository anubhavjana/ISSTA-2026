## Abstract

Plugin-based software frameworks allow hardware accelerator vendors to extend PyTorch without modifying its source. Yet this modularity comes at a hidden cost. PyTorch's test suite spans hundreds of thousands of test instances, yet remains largely inaccessible to out-of-tree (OOT) backends. The reason: pervasive first-party device assumptions integrated into the tests. For a new accelerator, rewriting this suite from scratch is infeasible. Running it unmodified is equally unrealistic. The real milestone is using the same infrastructure that defines correctness for first-party PyTorch devices. Without systematic tooling to achieve this, plugin developers are forced into manual, error-prone test curation. This approach neither scales with the PyTorch codebase nor survives its evolution. We present Discover, Adapt, Reuse -- an automated, repository-memory-aware test-driven enablement and adaptation framework for third-party PyTorch plugins. It addresses three stages of the adaptation lifecycle. (a) A Test Discovery stage uses static and dynamic code path analysis. It combines this with a graph-structured repository memory and an agentic search over PyTorch and plugin codebases. Together, these identify the minimal set of PyTorch tests that exercise plugin-relevant code paths without requiring exhaustive execution. (b) A Test Adaptation stage applies a novel declarative configuration format. This format captures plugin-aware patch semantics: dtype exclusions, decorator substitutions, skip policies, forced expected-failure annotations, and per-invocation CPU tensor relocation. Adaptations are applied on-the-fly at test collection time, without modifying PyTorch source files. (c) A Test Refactoring stage employs an LLM-based coding agent. The agent is guided by manually verified refactoring exemplars and a structured feedback loop from an execution harness. It automatically transforms device-specific test code such as CUDA-hardcoded decorators, device branches, and API calls into backend-agnostic equivalents compatible with the plugin. Unlike prior work on test selection, suite minimization, or metamorphic test generation, our approach reasons jointly over PyTorch and the plugin repository, which operate on different ownership and release cycles. It produces adaptations that remain composable and auditable as both evolve. We demonstrate the framework on a real-world OOT accelerator plugin in the open-source IBM torch-spyre. There, it substantially reduces plugin test maintenance burden while preserving PyTorch's test quality standards.
